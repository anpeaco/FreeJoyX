# Layer 12 — The two boards: F103 vs F411

FreeJoyX runs on two chips. Most of the code is shared, but the low-level hardware
access is written twice. Understanding the seam between "shared" and "per-board"
makes the `#ifdef`s stop being noise.

## The two targets

| | BluePill | BlackPill |
|---|---|---|
| Chip | STM32**F103**C8T6 | STM32**F411**CEU6 |
| Core | Cortex-M3 | Cortex-M4 (with FPU) |
| Flash | 64 KB | 512 KB |
| SRAM | 20 KB | 128 KB |
| Clock | 72 MHz | 96 MHz (HSE-PLL) |
| Vendor lib | StdPeriph (legacy) | HAL + LL (Cube) |
| USB stack | USB FS Device lib | Cube USBD CustomHID |
| Build flag | `BOARD_F103_BLUEPILL` | `BOARD_F411_BLACKPILL` |

The F411 port is the fork's headline feature — upstream FreeJoy is F103-only, and
"add F411/F401 support" is the single most-requested upstream issue.

## What's shared vs per-board

**Shared (board-independent):** the application logic — `analog.c` pipeline,
`buttons.c`, `encoders.c`, the config model, the wire format, the HID report shapes.
This is the bulk of the code and it doesn't care which chip it's on.

**Per-board (the BSP — Board Support Package):** anything that touches registers or
chip-specific peripherals. The fork introduced a **BSP seam**: board-specific code
lives under `board/f103_bluepill/` and `board/f411_blackpill/`, behind `Board_*`
functions called by the shared code. Examples you've seen:

- `Board_RelocateVectorTable()` — different VTOR offset per board.
- `Board_ClockInit_F411()` — the 96 MHz recipe.
- `Board_USB_Init()` / `Board_USB_DeInit()` / `Board_EnterDfu()` — wrap the two
  different USB stacks behind one interface.
- `Board_GetSerialNum()`, `Board_TickStop()`, etc.

So shared code says `Board_USB_Init()`; the build links in the right implementation
for the target.

## Where the `#ifdef`s live

Two patterns:

1. **BSP seam (preferred)** — the difference is hidden behind a `Board_*` function;
   shared code has no `#ifdef`. This is the direction the fork is moving.
2. **Inline `#ifdef BOARD_*`** — where the difference is small and local, the two
   code paths sit side by side. The clearest example is `analog.c::AxesInit()` /
   `ADC_Conversion()`: the F103 path uses StdPeriph (`ADC_Init`, `DMA_Init`); the
   F411 path pokes `ADC1->SQR1`, `DMA2_Stream4->CR`, etc. directly.

## Chip differences that have actually bitten

These make good cautionary examples (each was a real bug):

- **DMA transfer size** (#24) — F411 ADC DMA was set to *word* not *halfword*, so a
  second axis read as 0 → stuck at −32767. Fixed by using CMSIS field macros
  (`DMA_SxCR_MSIZE_0`) instead of magic `<<` shifts.
- **SRAM validity mask** — the bootloader's "is the app valid?" check masks the
  stack pointer against the SRAM range; F103 uses `0x2FFE0000`, F411 uses a
  128 KB range. Wrong mask → bootloader won't jump.
- **HID descriptor padding** (#2) — F411 Cube CustomHID reported a padded descriptor
  length; Windows rejected it (Code 10). Fixed with an exact-size `_Static_assert`.
- **Backup register** — F103 stores the DFU magic in `BKP->DR4`; F411 in
  `RTC->BKP0R`. Different register, same idea.
- **Halfword flash program** — both program flash in halfwords in the DFU path, but
  via different APIs (`FLASH_ProgramHalfWord` vs `HAL_FLASH_Program`).

## The takeaway

When you read shared code, assume it's board-agnostic. When you hit a `Board_*`
call or a `#ifdef BOARD_*`, that's the seam — the place where the same idea is
expressed in two chips' dialects. Bugs cluster at that seam, because it's where the
"write it twice" risk lives.
